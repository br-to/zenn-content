## はじめに

こんにちは！ブロックチェーン×AI Agentで自律経済圏を創るKomlock labでエンジニアをしている小原です。

[OpenClaw](https://github.com/openclaw/openclaw)というオープンソースのAIエージェントフレームワークを使い始めて約1ヶ月。Discord上でマルチエージェント構成を組み、予測市場の監視ツールやニュース自動収集を運用しています。

この記事では、実際に運用する中で見つけたユースケース、便利だったTips、そしてやらかした失敗談をまとめます。

## OpenClawとは

OpenClawは、LLMを使ったパーソナルAIエージェントをDiscord・Telegram・Slackなどのメッセージングプラットフォーム上で動かせるフレームワークです。

https://github.com/openclaw/openclaw

特徴的なのは以下の点です。

- ワークスペースファイル（SOUL.md, MEMORY.md等）でエージェントの人格や記憶を定義できる
- cronジョブで定期タスクを実行できる
- 複数エージェントをチャンネルごとに配置できる
- スキルやツール（シェル実行、Web検索等）を組み合わせて自律的に動かせる

## ユースケース1: 予測市場のオッズ監視

### やりたかったこと

Polymarketの予測市場オッズが急変動したとき、いち早く気づきたい。ブラウザを開いて手動チェックするのは非現実的なので、自動化したい。

### 構成

```
cron(3時間ごと) → odds_scanner.js
  → Polymarket CLIで24カテゴリの市場を検索
  → 前回のオッズ(JSON)と比較
  → 3%以上変動 → correlation_tracker.js(関連資産の価格記録)
                → OpenClaw経由でDiscord通知
                → AIエージェントがニュース検索＋分析
```

ポイントは、スキャン部分にLLMを使っていないこと。Node.jsスクリプト + Polymarket CLIだけで動くのでコストゼロです。AIエージェントが動くのはアラートが出たときだけ。

### 実装のキモ

odds_scanner.jsが毎回JSONファイルに最新のオッズを保存して、次回実行時に前回との差分を取る。シンプルだけど確実に動きます。

```js
// 前回との比較
if (history[mid]) {
  const prevPct = history[mid].yes_pct ?? yesPct;
  const delta = yesPct - prevPct;
  if (Math.abs(delta) >= ALERT_THRESHOLD) {
    alerts.push({ /* ... */ });
  }
}
```

監視カテゴリはクリプト、AI、金融政策、地政学など24キーワード。広く浅く張っておいて、変動があったものだけ深掘りする設計です。

https://github.com/br-to/polymarket-monitor

### 実際に検知できた例

イラン停戦のオッズが57.5%→22.5%に急落したとき、通常のニュースサイトより先にアラートが飛んできました。予測市場のオッズはニュースより速いことがあるので、この監視は情報収集ツールとして有用です。

## ユースケース2: マルチエージェント構成

### 4体のエージェント

Discordのチャンネルごとに役割の異なるエージェントを配置しています。

- **メインエージェント**（Claude Opus）: 技術的な議論、コードレビュー、意思決定のサポート
- **記事校正エージェント**（Claude Sonnet）: Zenn記事の文体調整
- **ニュース収集エージェント**（Claude Sonnet）: 1日3回、ブロックチェーン×AIニュースを自動収集
- **市場分析エージェント**（Claude Sonnet）: 予測市場のアラート分析

### コスト最適化

ここで重要なのが、メインだけOpusで残りはSonnetにしていること。OpenClawではチャンネルごとにモデルを切り替えられるので、高度な判断が必要な場面はOpus、定型作業はSonnetという使い分けが可能です。

```json
{
  "channelOverrides": {
    "discord:1480358078070390864": {
      "model": "amazon-bedrock/us.anthropic.claude-sonnet-4-5-20250929-v1:0"
    }
  }
}
```

## ユースケース3: twitter-cliでX上の話題を拾う

ニュース収集をBrave Search APIだけに頼っていたのですが、Xで話題になっているネタを取りこぼしていることに気づきました。

そこで、[twitter-cli](https://github.com/keiidev/twitter-cli)というCookie認証ベースの非公式CLIツールを使って、X上のバズりポストも拾うようにしています。

```bash
# Topフィルタで話題のポストだけ取得
twitter search "Polymarket" --json --max 10 -t Top
```

これをニュース収集エージェントのcronに組み込んで、1日3回の収集時にXの検索結果も含めるようにしました。Web検索では出てこない速報や、エンジニア界隈のリアルな反応が拾えるのが良いです。

注意点として、twitter-cliは非公式APIを使っているため、利用規約的にはグレーゾーンです。読み取り専用で使うのが無難です。

## VPSだからできること

OpenClawをVPS上で24時間動かしているからこそ実現できていることがいくつかあります。

**cronが止まらない**

ローカルPCだとスリープや再起動でcronが止まります。VPS上で動かしているので、夜中の3時でも予測市場のスキャンが走り、アラートが飛んできます。朝起きてDiscordを開いたら「深夜にイラン停戦オッズが急落した」という分析が届いている、という体験は、VPSならではです。

**エージェント間の連携が途切れない**

マルチエージェント構成で、あるエージェントが検知した情報を別のエージェントに渡して分析させる、というフローを組んでいます。これがローカルだと、PCを閉じた瞬間に途切れます。VPSなら24時間連携し続けます。

**自分が寝ている間も働いてくれる**

これが地味に一番大きい。深夜にニュースが出たとき、自分は寝ていてもエージェントがキャッチして翌朝までにまとめてくれる。人間が24時間張り付く必要がなくなります。

VPSのコストは月数千円ですが、それ以上の価値があると感じています。

## Tips

### メモリは2段階で管理する

OpenClawのエージェントはセッションごとにコンテキストがリセットされます。記憶を持続させるには、ワークスペースファイルに書き込む必要があります。

うちでは以下の2段階で管理しています。

- **日次ログ** (`memory/YYYY-MM-DD.md`): その日の生のメモ。会話中に重要情報が出たらその場で書き込む
- **長期メモリ** (`MEMORY.md`): 日次ログから抽出した、長期的に残す価値のある情報

さらに、毎日0:00に自動でMEMORY.mdを整理するcronジョブを設定しています。日次ログを読んで、重要な情報を長期メモリに移し、古くなった情報を削除する。これでMEMORY.mdが肥大化してコンテキストを圧迫するのを防いでいます。

### cronのtzを忘れない

OpenClawのcronジョブを設定するとき、`tz`（タイムゾーン）を指定しないとUTCで実行されます。

```bash
# NG: UTCの15時 = JSTの0時に実行されてしまう
openclaw cron add --cron "0 15 * * *" --message "..."

# OK: JSTの15時に実行される
openclaw cron add --cron "0 15 * * *" --tz "Asia/Tokyo" --message "..."
```

実際にこれでニュース収集が深夜に走ってしまい、しばらく気づかなかったことがあります。

### コンパクション対策はバージョンアップ

セッションが長くなるとコンテキストが圧縮（コンパクション）されて、以前の会話内容を忘れることがあります。v2026.3.8でこの問題が改善されたので、同じ症状がある方はアップデートをおすすめします。

## 失敗談

### API課金で$5.72溶かした話

X（Twitter）APIの共通フォロワーを調べようとして、コスト計算をせずに大量のAPIコールを実行。ユーザーデータは1件あたり$0.01の課金で、558件取得して$5.72消費しました。

教訓は明確です。

1. 有料APIを叩く前にコストを見積もる（1リクエストいくら × 何回 = 合計いくら）
2. `max_results`は最小値でテストしてから拡大する
3. スクリプトにはAPI呼び出し回数のリミットを必ず入れる

AIエージェントに「調べて」と雑に投げると、コスト意識なく大量にAPIを叩くことがあります。AGENTS.mdにルールとして明記することで再発を防いでいます。

### 閾値の試行錯誤

予測市場の監視で、アラートの閾値設定に苦労しました。

- **0.1%**: アラートが多すぎて通知の嵐
- **5%**: 静かな市場ではほとんど引っかからない
- **3%**: ちょうどいい

最適な閾値は市場によって異なるので、運用しながら調整するしかありません。

## まとめ

OpenClawは「AIエージェントにDiscord上で仕事をさせる」フレームワークとして、かなり実用的です。特に以下のパターンと相性がいいと感じました。

- 定期的な情報収集・監視
- 複数エージェントの役割分担
- ファイルベースの記憶管理

一方で、メモリ管理やcron設定など、運用してみないとわからない落とし穴もあります。この記事が、これからOpenClawを使う方の参考になれば嬉しいです。
